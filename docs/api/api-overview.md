# API overview

All application endpoints are versioned under `/api/v1`. The operational endpoints are `GET /health`, `GET /ready`, and `GET /api/v1/version`.

Implemented first-slice endpoints: `GET|POST /api/v1/sync/managers`, `POST /api/v1/ingestion/uploads`, `GET /api/v1/catalog/snapshots`, `POST /api/v1/catalog/snapshots/{id}/status`, and `POST /api/v1/catalog/snapshots/{id}/activate`. Catalog uploads now persist immutable raw-import provenance, validation-run IDs, release IDs, configuration and input hashes, object counts, and validation summaries alongside the staged snapshot. Duplicate `release_id` values are idempotent only when they point to the same `source_hash`; conflicting content is rejected with `409 RELEASE_ID_HASH_CONFLICT`. Snapshot lifecycle is constrained to `staged -> reviewed -> published -> superseded`, with publishing reserved for the activate workflow, and large artifact fields such as embedded APK blobs are stored as manifest-only evidence rather than inline snapshot payload. Authentication endpoints are the next required security milestone and must gate all user data before deployment.
